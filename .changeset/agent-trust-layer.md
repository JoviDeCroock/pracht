Add the agent trust layer on top of the capability core: verified agent identity, effect-class enforcement, and an eval harness (see docs/AGENT_TRUST.md).

- `@pracht/core` implements Web Bot Auth verification (RFC 9421 HTTP Message Signatures per draft-meunier-web-bot-auth-architecture-02, Ed25519 via WebCrypto, key resolution from statically configured keys or allowlisted `/.well-known/http-message-signatures-directory` JWKS directories with a TTL cache — fail closed everywhere). Opt in via `defineApp({ agents: { webBotAuth } })`; the verified identity surfaces as `context.agent` for middleware, loaders, API routes, and capability `run()`, with `"observe"`/`"require"` policy modes (per-app default, per-capability `agentPolicy` override — unverified requests to `"require"` capability endpoints get a `401 agent_required` envelope). Every capability dispatch (HTTP and `invokeCapability()`) now emits a structured audit event via `setCapabilityAuditHook()` or the `onCapabilityAudit` runtime option.
- Destructive capabilities can now be exposed over HTTP behind a server-verified prepare/commit confirmation flow: the first call returns `409 confirmation_required` with a short-lived HMAC token bound to principal + capability + canonicalized input; a second call with identical input and the `x-pracht-confirm` header runs it, and tampered/expired/mismatched tokens fail closed with 403. Requires `PRACHT_CONFIRMATION_SECRET` (or `setCapabilityConfirmationSecret()`); optional best-effort in-memory single-use cache. `expose.webmcp`/`expose.mcp` for destructive capabilities stays disallowed — `@pracht/capabilities` and the registry enforce the new rule.
- `@pracht/cli` adds `pracht eval [files...]`: a scripted agent-task harness that runs `evals/**/*.eval.json` scenarios against the capability HTTP projection with `$steps[n].<path>` references (e.g. carrying the confirmation token), subset output matching, a human transcript or `--json`, and exit code 1 on failed expectations. `pracht verify` now checks the destructive exposure rules (agent projections forbidden; HTTP exposure requires the confirmation secret).
