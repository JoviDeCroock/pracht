# Pages Router

The pages router is an optional file-system routing mode for viact. When enabled, it scans a `pages/` directory and auto-generates a `routes.ts` file — no manual route wiring needed.

This is ideal for simpler apps or for migrating from Next.js pages router.

## Setup

Enable the pages router in your Vite config:

```ts
import { viact } from "@viact/vite-plugin";

export default {
  plugins: [
    viact({
      pages: true, // uses defaults: dir="/src/pages", defaultRender="ssr"
    }),
  ],
};
```

Or with custom options:

```ts
viact({
  pages: {
    dir: "/src/pages", // directory to scan (default: "/src/pages")
    defaultRender: "ssg", // fallback render mode (default: "ssr")
    outFile: "/src/routes.generated.ts", // generated file path (default)
  },
});
```

## File Conventions

| File                    | Route            |
| ----------------------- | ---------------- |
| `pages/index.tsx`       | `/`              |
| `pages/about.tsx`       | `/about`         |
| `pages/blog/index.tsx`  | `/blog`          |
| `pages/blog/[slug].tsx` | `/blog/:slug`    |
| `pages/[...all].tsx`    | `/*` (catch-all) |
| `pages/_app.tsx`        | Shell (layout)   |

- Only **one** `_app.tsx` is supported, at the root of the pages directory
- Files prefixed with `_` (other than `_app`) are ignored
- Directories prefixed with `_` are ignored
- Supported extensions: `.tsx`, `.ts`, `.jsx`, `.js`, `.md`, `.mdx`

## Shell (`_app.tsx`)

The `_app.tsx` file is your layout wrapper. It works like a viact shell:

```tsx
// src/pages/_app.tsx
import type { ShellProps } from "viact";

export function Shell({ children }: ShellProps) {
  return (
    <div class="app-layout">
      <nav>...</nav>
      <main>{children}</main>
    </div>
  );
}

export function head() {
  return { title: "My App" };
}
```

## Render Mode

Each page can declare its render mode via an exported constant:

```tsx
// src/pages/about.tsx
export const render = "ssg";

export function Component() {
  return <h1>About</h1>;
}
```

Supported values: `"ssr"`, `"ssg"`, `"spa"`, `"isg"`

If omitted, the page uses the `defaultRender` from your plugin config (defaults to `"ssr"`).

## Data Loading

Pages support loaders and actions just like regular viact routes:

```tsx
// src/pages/blog/[slug].tsx
import type { LoaderArgs, RouteComponentProps } from "viact";

export const render = "ssr";

export async function loader({ params }: LoaderArgs) {
  const post = await fetchPost(params.slug);
  return { post };
}

export function Component({ data }: RouteComponentProps<typeof loader>) {
  return <article>{data.post.title}</article>;
}
```

## Generated Output

The plugin writes a `routes.generated.ts` file that uses the standard viact routing API. You can inspect this file to understand exactly what routes are generated.

Example output:

```ts
// Auto-generated by viact pages router — do not edit manually.
import { defineApp, route, group } from "viact";

export const app = defineApp({
  shells: {
    root: "./src/pages/_app.tsx",
  },
  routes: [
    group({ shell: "root" }, [
      route("/", "./src/pages/index.tsx", { id: "index", render: "ssr" }),
      route("/about", "./src/pages/about.tsx", { id: "about", render: "ssg" }),
      route("/blog/:slug", "./src/pages/blog/[slug].tsx", { id: "blog-slug", render: "ssr" }),
    ]),
  ],
});
```

## Ejecting

To switch from pages router to manual routing:

1. Remove the `pages` option from your viact plugin config
2. Rename `routes.generated.ts` to `routes.ts`
3. Edit `routes.ts` as needed — it's standard viact code

## Dev Server

In development, the pages router watches your pages directory for changes:

- **New file added**: routes regenerate automatically
- **File deleted**: routes regenerate automatically
- **Render mode changed**: routes regenerate on save

## Migrating from Next.js Pages Router

The viact pages router uses the same file conventions as Next.js, making migration straightforward.

### Step-by-step

1. **Copy your `pages/` directory** into `src/pages/`

2. **Update `_app.tsx`** — Next.js uses `Component`/`pageProps`, viact uses `Shell`/`children`:

   ```tsx
   // Next.js
   export default function App({ Component, pageProps }) {
     return (
       <Layout>
         <Component {...pageProps} />
       </Layout>
     );
   }

   // viact
   export function Shell({ children }: ShellProps) {
     return <Layout>{children}</Layout>;
   }
   ```

3. **Replace data fetching**:

   | Next.js              | viact                                                            |
   | -------------------- | ---------------------------------------------------------------- |
   | `getServerSideProps` | `export async function loader()` (render is `"ssr"` by default)  |
   | `getStaticProps`     | `export async function loader()` + `export const render = "ssg"` |
   | `getStaticPaths`     | `export function prerender()` (returns param list)               |

4. **Replace `useRouter`** with viact's routing:

   ```tsx
   // Next.js
   import { useRouter } from "next/router";
   const { query } = useRouter();

   // viact — params come from the loader or component props
   export function Component({ params }: RouteComponentProps) {
     // params.slug, etc.
   }
   ```

5. **Replace `next/link`** with viact's `Link`:

   ```tsx
   // Next.js
   import Link from "next/link";
   <Link href="/about">About</Link>;

   // viact
   import { Link } from "viact";
   <Link href="/about">About</Link>;
   ```

6. **Replace `next/head`** with the `head()` export:

   ```tsx
   // Next.js
   import Head from "next/head";
   <Head>
     <title>Page Title</title>
   </Head>;

   // viact
   export function head() {
     return { title: "Page Title", meta: [{ name: "description", content: "..." }] };
   }
   ```

7. **Dynamic routes** work the same way — `[param].tsx` and `[...slug].tsx` are supported with identical bracket syntax.

### What's different

- No `getInitialProps` — use `loader()` instead
- No `_document.tsx` — viact handles the HTML document
- No API routes in the pages directory — use viact's `src/api/` directory instead
- Render mode is explicit per-page via `export const render` rather than choosing between `getServerSideProps` vs `getStaticProps`
